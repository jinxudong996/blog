##### 一、为啥会有微前端

在企业级前端开发中，尤其是管理后台型应用，随着业务的不断扩展，单体式前端架构逐渐暴露出诸多问题。早期为了快速交付，很多公司会选择低代码平台或统一的 SPA 技术栈来搭建系统，这在初期确实提高了效率，但当业务体量变大后，问题就来了：新模块必须依赖原平台开发，技术栈被锁死；不同业务之间无法灵活集成，体验割裂；每次改动都需要全量打包和发布，成本极高；性能优化和重构也因耦合严重而举步维艰。尤其是在多团队协作的情况下，一个仓库中往往塞满了权限管理、视频监控、地图、数据可视化等多个模块，代码互相影响，版本冲突频发，升级一次框架就像“全员停工大修”。

微前端的价值，正是要把这样“长在一起的应用”拆成“可以独立生长的模块”。它允许不同团队使用不同的技术栈各自开发、独立部署，同时在最终呈现时仍保持一致的 SPA 体验，而不是像 iframe 那样割裂。对于历史遗留系统，也可以采用渐进式重构方式：旧模块继续运行，新模块逐步替换，无需一次性推倒重来。可以说，微前端并不是为了追求“炫技”，而是为了解决前端在复杂业务环境下**难以维护、难以扩展、难以协作的问题**。当你发现一个系统已经复杂到“想改又不敢改”的时候，微前端往往就是破局之道。

##### 二、方案了解

###### iframe方案

iframe是非常常见的微前段设计方案之一，   接下来首先介绍下浏览器的多进程架构设计。

在打开浏览器访问web应用时，多个标签页之间不会互相收到彼此的影响，比如某个标签页所在的应用崩溃了，其他应用也不会受到影响，这就是浏览器的多进程架构设计的直接体现。

以chrome浏览器为例，在运行时会常驻在Browser主进程，打开标签页就会动态的创建Renderer进程，

- **Browser 主进程**：主要负责处理网络资源请求、用户的输入输出 UI 事件、地址栏 URL 管理、书签管理、回退与前进按钮、文件访问、Cookie 数据存储等。Browser 进程是一个常驻的主进程，它也被称为代理进程，会派生进程并监督它们的活动情况。除此之外，Browser 进程会对派生的进程进行沙箱隔离，具备沙箱策略引擎服务。Browser 进程通过内部的 I/O 线程与其他进程通信，通信的方式是 [IPC](https://link.juejin.cn/?target=https%3A%2F%2Fwww.chromium.org%2Fdevelopers%2Fdesign-documents%2Finter-process-communication%2F) & [Mojo](https://link.juejin.cn/?target=https%3A%2F%2Fchromium.googlesource.com%2Fchromium%2Fsrc%2F%2B%2FHEAD%2Fmojo%2FREADME.md)。
- **Renderer 进程**：主要负责标签页和 iframe 所在 Web 应用的 UI 渲染和 JavaScript 执行。Renderer 进程由 Browser 主进程派生，每次手动新开标签页时，Browser 进程会创建一个新的 Renderer 进程。

web应用在Renderer进程中，浏览器为了提升安全性，需要通过常驻的Browser主进程对Renderer进程进行沙箱隔离设计，从而实现web应用进行隔离和管控。

在微前端中使用iframe来实现，需要一个主应用，包含导航和内容区的设计，通过切换导航来控制内容区微应用 A / B / C 的加载和卸载。而不同的jframe之间，可以通过postMessage来实现通信。
