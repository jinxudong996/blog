# 从文本到可镜头级编辑短剧：LangChain 项目学习路线

## 1. 学习目标

最终产品建议拆成两个边界清晰、通过结构化数据连接的系统：

1. **文本转短剧剧本系统**：小说、故事梗概或用户创意 -> 人物小传 -> 剧情大纲 -> 分集 -> 分场 -> 分镜脚本。
2. **短剧剧本转短剧系统**：分镜脚本 -> 角色/场景/道具资产 -> 镜头生成 -> 配音/音效/字幕 -> 合成与导出。

“可以镜头级更改”不是最后再补的功能，而是整个系统的数据模型。从学习阶段开始，每个镜头就应该有稳定 ID、结构化属性、生成状态和版本。

## 2. 最终系统的核心数据流

```text
SourceText
  -> StoryBible（世界观、人物、规则）
  -> SeriesOutline
  -> EpisodeScript[]
  -> Scene[]
  -> Shot[]
  -> MediaAsset[]
  -> Timeline
  -> RenderedEpisode
```

最小 `Shot` 模型建议至少包含：

```json
{
  "shot_id": "ep01-sc03-sh05",
  "scene_id": "ep01-sc03",
  "duration_seconds": 4.0,
  "shot_size": "close_up",
  "camera_movement": "slow_push_in",
  "characters": ["char_lin_xia"],
  "location": "loc_old_apartment",
  "action": "林夏看到桌上的离婚协议，手指停在签名处",
  "dialogue": "原来你早就准备好了。",
  "visual_prompt": "...",
  "negative_prompt": "...",
  "audio_cues": ["rain", "paper_rustle"],
  "continuity_refs": ["ep01-sc03-sh04"],
  "asset_ids": [],
  "status": "draft",
  "version": 1
}
```

后续修改某个镜头时，只重新执行该 `shot_id` 及被它影响的下游节点，不重新生成整部短剧。

## 3. 教程项目总览

| 阶段 | 项目 | 主要技术 | 与最终目标的关系 |
| --- | --- | --- | --- |
| LCEL | 1. 故事结构化抽取器 | Prompt、Model、Parser、Pydantic | 把任意文本变成稳定数据 |
| LCEL | 2. 多链剧本工厂 | RunnableSequence/Parallel/Branch | 完成从故事到分场剧本 |
| LCEL | 3. 分镜脚本生成与质检 | 结构化输出、重试、回退、流式输出 | 建立镜头级数据契约 |
| RAG | 4. 编剧知识库 | Loader、Splitter、Embedding、Retriever | 检索类型规则和优秀范例 |
| RAG | 5. 长文本事实与连续性助手 | 元数据过滤、混合检索、重排 | 避免长篇改编中人设和道具穿帮 |
| LangGraph | 6. 可循环修订的编剧 Agent | State、Node、Edge、Checkpoint | 实现生成-审核-定向重写 |
| LangGraph | 7. 镜头生成调度器 | 并行节点、持久化、人工介入 | 将单镜头转换为视频制作任务 |
| 综合 | 8. 镜头级短剧工作台 MVP | FastAPI、任务队列、对象存储、时间线 UI | 打通两个系统的最小闭环 |

## 4. 项目一：故事结构化抽取器

### 目标

输入一段故事或小说片段，输出经过 Pydantic 校验的 `StoryBible`，包含世界观、人物、人物关系、事件和未解冲突。

### 教程章节

1. 用 `ChatPromptTemplate` 区分 system/human 消息。
2. 用 Pydantic 定义 `Character`、`Relationship`、`StoryEvent` 和 `StoryBible`。
3. 使用 `with_structured_output` 得到可校验数据。
4. 用 LCEL 组装 `prompt | model | parser`，比较 `invoke`、`batch` 和 `stream`。
5. 处理缺失字段、冲突事实和幻觉信息。
6. 为固定输入准备预期 JSON，编写最小回归测试。

### 验收成果

- 命令行可读取 `.txt` 并生成 `story_bible.json`。
- 输出不依赖 Markdown 或字符串切割。
- 未知信息显式为 `null` 或空列表，不自行补全。

## 5. 项目二：多链剧本工厂

### 目标

将 `StoryBible` 转换为一集 1-3 分钟的短剧剧本，体验不同 Runnable 的组合方式。

### 工作流

```text
故事解析
  -> 类型路由（爱情/悬疑/喜剧/其他）
  -> 主线与副线规划
  -> 人物弧光 + 节拍表（并行）
  -> 分场剧本
  -> 最终格式化
```

### 教程章节

1. 用 `RunnableLambda` 进行输入适配和数据归一化。
2. 用 `RunnableParallel` 并行生成人物弧光和剧情节拍。
3. 用 `RunnableBranch` 对不同类型选择不同提示词。
4. 使用 `assign`、`pick`、`passthrough` 管理中间数据。
5. 用 `with_config` 记录剧集 ID、提示词版本和追踪标签。
6. 为每条链分别测试，再测试组合后数据契约。

### 验收成果

- 输出 `episode_script.json` 和便于人阅读的 `episode_script.md`。
- 每个场景有 `scene_id`、内/外景、时间、地点、人物、行动、对白和剧情目标。
- 替换任一子链时，其他链无需修改。

## 6. 项目三：分镜脚本生成与质检

### 目标

把分场剧本转为 `Shot[]`，同时建立可恢复的模型调用和自动质检。

### 教程章节

1. 定义镜别、机位、运镜、时长、台词、视觉提示词等枚举与模型。
2. 按场景 `batch` 生成分镜，控制并发数与限流。
3. 为临时网络错误和解析错误配置 `with_retry`。
4. 为主模型配置 `with_fallbacks`，保存真实失败原因。
5. 建立确定性规则：镜头 ID 唯一、时长为正数、台词时长可容纳、角色必须存在。
6. 建立 LLM 审核链：构图可拍性、镜头重复、节奏、情绪连续性。
7. 使用流式事件展示当前处理到的场景和镜头。

### 验收成果

- 产出通过 Schema 和确定性规则的 `shots.json`。
- 审核结果定位到具体 `shot_id`，而不是只给全局评语。
- 可仅重新生成一个失败镜头。

## 7. 项目四：编剧知识库 RAG

### 目标

建立一个小型编剧知识库，在生成剧本时引用节拍、类型、对白和分镜规则，不直接模仿完整作品。

### 教程章节

1. 加载 Markdown、PDF 和结构化剧本范例。
2. 比较固定长度、标题感知和“场/镜头”语义分块。
3. 生成 embedding，保存文档类型、题材、来源和授权等元数据。
4. 构建向量检索器，实验 `top_k`、相似度阈值和 MMR。
5. 使用 query rewriting 将“这场不够紧张”转为可检索问题。
6. 生成答案时返回来源 ID，并对无证据问题拒绝伪造引用。
7. 构造 20-50 个标注问题，评估检索命中率和回答忠实度。

### 验收成果

- 能回答“60 秒悬疑短剧如何设置开场钩子”并显示来源。
- 可用题材、内容类型、授权范围过滤。
- 有可重复执行的离线评估脚本。

## 8. 项目五：长文本事实与连续性助手

### 目标

让系统在改编长小说时找回正确原文事实，并检查人物、时间、场所、服装、道具和人物关系的连续性。

### 教程章节

1. 将原文分别建成“原文片段索引”和“事实卡片索引”。
2. 抽取事件的人物、时间、地点、原文范围和置信度。
3. 使用元数据过滤限制某章、某人物或某时间线。
4. 组合 BM25 和向量检索，解决专有名词与语义问题。
5. 用 reranker 重排，用 parent document retrieval 找回完整上下文。
6. 在生成新场景前检索相关事实，生成后再运行连续性检查。
7. 输出 `ContinuityIssue[]`，每条问题带证据和建议修改的 `scene_id/shot_id`。

### 验收成果

- 能发现“上一镜杯子已打碎，下一镜又出现完好杯子”。
- 每项问题均带原文或前序镜头证据。
- 将“知识库规则”与“当前项目事实”分库管理。

## 9. 项目六：可循环修订的编剧 LangGraph Agent

### 目标

将 LCEL 链升级为有状态、可循环、可中断、可恢复的编剧工作流。

### 状态设计

```text
ScriptState = {
  project_id,
  source_text,
  story_bible,
  episode_outline,
  scenes,
  shots,
  review_issues,
  revision_count,
  approved,
  error
}
```

### 图流程

```text
analyze_story -> plan_episode -> write_scenes -> create_shots
                                             -> review
review --通过--> human_approval -> END
review --未通过--> route_issue -> revise_scene/revise_shot -> review
```

### 教程章节

1. 定义 State、Reducer、Node 和普通 Edge。
2. 用 Conditional Edge 将问题路由到场景或镜头修订节点。
3. 设置最大修订次数，避免无限循环。
4. 用 Checkpointer 按 `thread_id` 保存和恢复项目。
5. 在大纲审批和最终审批前加入 human-in-the-loop interrupt。
6. 编辑人工意见后从中断点恢复，而非从头执行。
7. 流式输出节点事件，在界面上显示实时进度。

### 验收成果

- 程序中断后可从 checkpoint 恢复。
- 人工否决某镜头后，只进入定向修订分支。
- 所有修订有原始版本、修订意见和新版本可追溯。

## 10. 项目七：镜头生成调度器

### 目标

先使用可替换的假生成器（mock provider）理解任务编排，再接入真实的图片、视频、TTS 和音频服务。

### 每个镜头的子图

```text
resolve_assets
  -> build_visual_prompt
  -> generate_keyframe
  -> human_review_keyframe
  -> generate_video
  -> generate_voice + generate_sfx（并行）
  -> compose_shot
  -> quality_check
```

### 教程章节

1. 为图像、视频、TTS 提供商定义统一 adapter 接口。
2. 将每个镜头表示为独立任务，根据依赖并行调度。
3. 保存外部 job ID，使用轮询或 webhook 更新任务状态。
4. 设计幂等键，防止重试时重复扣费或重复生成。
5. 将中间产物保存到对象存储，数据库只保存 URI 和元数据。
6. 修改镜头时计算影响范围：只重生成视频，还是同时重生成语音与合成。
7. 在调用付费 API 前加入成本预估、额度和审批节点。

### 验收成果

- mock provider 可完整跑通，无需一开始就消耗真实 API 费用。
- 任意镜头失败不会丢失其他镜头的成果。
- 修改台词后，只标记该镜头的语音、字幕和合成产物过期。

## 11. 项目八：镜头级短剧工作台 MVP

### 目标

用一个简单 Web 应用打通两个系统，证明镜头级修改和增量重生成的完整闭环。

### 最小页面

1. **项目页**：创建项目、导入文本、查看处理状态。
2. **剧本编辑页**：左侧分集/分场树，中间剧本，右侧人物和连续性问题。
3. **分镜页**：按顺序展示镜头，可增删、拖动、拆分、合并和定向重生成。
4. **镜头属性面板**：编辑镜别、运镜、时长、角色、动作、台词、提示词和随机种子。
5. **时间线页**：视频、对白、音效、配乐和字幕轨道，支持预览与导出。

### 后端最小模块

```text
api/          HTTP/SSE/WebSocket 接口
domain/       Project/Script/Scene/Shot/Asset/Revision 模型
workflows/    LangGraph 图与 LCEL 链
providers/    LLM/Image/Video/TTS 适配器
repositories/ 数据库与对象存储
workers/      异步生成任务
evaluation/   检索、剧本和镜头评估
```

### 验收成果

- 导入一段文本，能通过人工审批生成分镜列表。
- 编辑单个镜头后创建新版本，原版本仍可恢复。
- 仅重新运行被修改镜头的必要生成节点。
- 可导出带字幕的演示视频；初期可使用静态分镜图 + TTS + FFmpeg 代替 AI 视频。

## 12. 建议学习顺序与节奏

| 周 | 主题 | 项目里程碑 |
| --- | --- | --- |
| 1 | LCEL 基础与结构化输出 | 完成项目一 |
| 2 | Runnable 组合、路由、并行 | 完成项目二 |
| 3 | 可靠性、流式输出、测试 | 完成项目三 |
| 4-5 | RAG 数据摄取与基础检索 | 完成项目四 |
| 6-7 | 高级检索与 RAG 评估 | 完成项目五 |
| 8-9 | LangGraph 状态、路由、持久化 | 完成项目六 |
| 10-11 | 外部媒体任务编排 | 完成项目七 |
| 12+ | Web 工作台与集成 | 完成项目八 MVP |

不建议完全学完所有 LangChain API 再开始做项目。每个阶段只学当前项目必需的 API，用可测试产物验证理解，再进入下一阶段。

## 13. 贯穿所有项目的工程要求

1. **结构化优先**：模块间传递 Pydantic/JSON 对象，Markdown 只用于展示。
2. **ID 稳定**：项目、剧集、场景、镜头、角色和资产均有稳定 ID。
3. **版本可追溯**：保存输入、模型、提示词版本、参数、输出和人工修改。
4. **确定性校验优先**：能用代码检查的问题不交给 LLM 判断。
5. **副作用隔离**：“生成内容”和“调用付费媒体 API”分成不同节点。
6. **可观测**：记录耗时、token、费用、失败原因和节点输入输出。
7. **评估先行**：为故事事实、检索、剧本和镜头分别准备小型固定数据集。
8. **版权与安全**：记录语料和资产的授权来源，对用户上传内容进行隔离和访问控制。

## 14. 当前最适合的下一步

你当前的代码已经涉及 Prompt/Model/Parser、结构化输出、`RunnableParallel`、`RunnableBranch`、重试和回退。因此建议直接开始**项目一**，但把练习数据从“招聘信息”换成“故事与人物”。

第一个迭代只做以下四件事：

1. 定义 `StoryBible` Pydantic 模型。
2. 读取一段 1000-3000 字的短故事。
3. 通过 LCEL 产出通过校验的 JSON。
4. 用 5 份固定故事样本做回归测试。

完成这一步后，再用项目二把该 JSON 组装成真正的剧本工作流。
